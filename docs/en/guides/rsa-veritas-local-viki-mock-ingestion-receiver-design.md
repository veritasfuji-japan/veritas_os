# RSA ↔ VERITAS Local V.I.K.I. Mock Ingestion Receiver Design

## 1. Purpose

This document defines the VERITAS-side design for receiving synthetic payloads from a local V.I.K.I. mock generator.

- This is not a runtime implementation.
- This is not a live V.I.K.I. connection.
- This is not a production API endpoint.
- This is a design note for local mock ingestion only.
- The goal is to prepare the VERITAS-side receiver behavior before implementing any local mock receiver or controlled integration test.

## 2. Current baseline

The current merged baseline includes:

- Static sandbox documentation.
- Four dedicated per-variant validation snapshots.
- Static fixture matrix.
- Sandbox reviewer index.
- Live V.I.K.I. integration design note.
- Live V.I.K.I. integration reviewer checklist.

The local V.I.K.I. mock generator now simulates the four established state flags:

- SAFE_PROCEED
- DENSITY_THROTTLED
- ALGORITHMIC_HUMILITY_ENGAGED
- DEFERRAL_ENGAGED

## 3. Boundary model

- The V.I.K.I. mock generator emits a synthetic RSA-compatible payload.
- VERITAS must treat the payload as untrusted until schema validation passes.
- VERITAS must not ingest V.I.K.I. internal reasoning.
- VERITAS must not execute hidden V.I.K.I. logic.
- VERITAS must consume only the emitted payload.
- VERITAS maps accepted payloads through RSASandboxPayload and evaluate_rsa_sandbox_signal().
- VERITAS audit output must redact raw upstream intent/action fields by default.

## 4. Local mock ingestion flow

Local V.I.K.I. mock generator
→ synthetic RSA-compatible JSON payload
→ VERITAS local mock ingestion receiver
→ JSON parse
→ schema validation
→ RSASandboxPayload construction
→ evaluate_rsa_sandbox_signal(payload)
→ VERITAS decision
→ audit entry
→ sandbox commit-state gate

Clarifications:

- This flow is local-only.
- It does not require production network connectivity.
- It does not use real KYC data.
- It does not use live LLM text.
- It does not add production commit authority.

## 6. Example mock payload

```json
{
  "rsa_status": "ALGORITHMIC_HUMILITY_ENGAGED",
  "trigger_source": "SRC_Incomplete_Context",
  "original_llm_intent": "Recommend_Transaction_Approval",
  "rsa_action_taken": "Execution_Suspended_Awaiting_Reality_Sync",
  "timestamp": "2026-05-20T23:01:35.876Z"
}
```

The timestamp is dynamically generated by the mock generator and should be treated as an input field subject to validation.

## 7. VERITAS-side schema validation rules

VERITAS must treat every incoming local mock payload as untrusted until validation passes.

Required fields:

- rsa_status
- trigger_source
- timestamp

Optional accepted fields:

- original_llm_intent
- rsa_action_taken

Accepted rsa_status values:

- SAFE_PROCEED
- DENSITY_THROTTLED
- ALGORITHMIC_HUMILITY_ENGAGED
- DEFERRAL_ENGAGED

Validation failure cases:

- invalid JSON
- missing required field
- null required field
- empty trigger_source
- unknown rsa_status
- invalid timestamp
- payload shape mismatch
- attempt to bypass RSASandboxPayload
- raw upstream content that cannot be safely redacted

## 8. Fail-closed behavior

VERITAS must not infer SAFE_PROCEED from missing, malformed, delayed, or invalid local mock payloads.

If the local mock payload is invalid, VERITAS must fail closed.

Expected failure behavior:

- continuation_decision: PAUSE_FOR_HUMAN_REVIEW
- reason_code: UPSTREAM_MOCK_PAYLOAD_INVALID or equivalent future error code
- sandbox_commit_state: SUSPENDED_NOT_COMMITTED
- required_next_action: REQUEST_VALID_SYNTHETIC_PAYLOAD_OR_HUMAN_REVIEW

For timeout or unreachable local mock generator:

- continuation_decision: PAUSE_FOR_HUMAN_REVIEW
- reason_code: UPSTREAM_MIDDLEWARE_OFFLINE or equivalent future error code
- sandbox_commit_state: SUSPENDED_NOT_COMMITTED
- required_next_action: REQUEST_HUMAN_REVIEW_OR_RETRY_WITH_VALID_UPSTREAM_STATE

Clarifications:

- Only a schema-valid SAFE_PROCEED emitted by the mock generator may continue toward normal bind-boundary evaluation.
- No payload / timeout / malformed payload must never be converted into SAFE_PROCEED.

## 9. Audit and redaction rules

VERITAS audit entries should preserve:

- rsa_status
- trigger_source
- timestamp
- VERITAS continuation_decision
- VERITAS reason_code
- VERITAS sandbox_commit_state

VERITAS audit entries must redact by default:

- original_llm_intent
- rsa_action_taken
- raw upstream reasoning
- V.I.K.I. internal reasoning
- chain-of-thought
- hidden model state

The audit trail should store deterministic state and decision data, not raw upstream reasoning.

## 10. Compatibility contract

The v1 compatibility contract remains unchanged:

- rsa_status
- RSASandboxPayload
- evaluate_rsa_sandbox_signal()
- upstream_signal_source = "RSA"

Additional constraints:

- Do not rename rsa_status to viki_status in this phase.
- Do not rename RSASandboxPayload to VIKIPayload in this phase.
- Any naming migration must be handled separately as a v2 migration.

## 11. What this design validates

- The local V.I.K.I. mock generator can be treated as an upstream synthetic payload producer.
- VERITAS-side ingestion can be designed without live V.I.K.I. connection.
- The four established state flags can be mapped through the existing VERITAS receiver contract.
- Schema validation and fail-closed behavior are defined before implementation.
- Audit redaction expectations are defined before implementation.
- The boundary between V.I.K.I. and VERITAS remains preserved.

## 12. What this design does not validate

- It does not implement the receiver.
- It does not add an API endpoint.
- It does not connect live V.I.K.I. middleware.
- It does not validate live V.I.K.I. internal reasoning.
- It does not validate network transport.
- It does not validate authentication or authorization.
- It does not process live LLM text.
- It does not process real KYC data.
- It does not implement production AML/KYC compliance.
- It does not provide regulatory approval.
- It does not provide legal advice.
- It does not change production runtime governance.

## 13. Recommended next PR after this note

After this design note is merged, the next safe PR should be either:

- local mock receiver test fixture plan
- local mock receiver implementation behind an explicit test-only guard
- live payload schema draft
- controlled integration threat model

Do not implement any production live V.I.K.I. connection yet.

Any implementation must be local-only, synthetic-data-only, and test-gated first.
