## 重要指摘（優先度順）

### P1: Startup 設定検証失敗時に起動継続する挙動（中〜高）
**観測**
- `startup` フック内の設定検証で例外を包括捕捉し、warning ログのみで起動継続する経路が存在する。

**リスク（セキュリティ警告）**
- 誤設定状態で API が継続稼働すると、認証・監査・通信設定の意図しない緩和を見逃す可能性がある。
- **これは運用時のセキュリティリスク**（防御の前提崩壊）として扱うべき。

**推奨対応**
- `VERITAS_ENV=prod` では fail-fast（例外再送出）を標準化する。
- `dev/stg` のみ warning 継続を許可し、環境ごとの分岐をコード上で明示する。
- SRE 向けに「起動失敗=設定不備」の運用ドキュメントを追記する。

**完了条件（DoD）**
- prod で設定不整合時にプロセスが確実に異常終了すること。
- dev/stg は従来どおり検証可能であること。
- 期待挙動を確認する自動テスト（起動時設定検証）を追加すること。

### P2: FastAPI `on_event` の非推奨 API 利用（中）
**観測**
- `@app.on_event("startup"|"shutdown")` 依存が残存しており、将来的に移行コスト増の要因となる。

**リスク**
- 直近の動作には影響しないが、Framework 更新時の互換性リスクと保守コストを押し上げる。

**推奨対応**
- `lifespan` コンテキストへ移行し、起動/停止処理を一本化する。
- 監視初期化・外部接続解放などの副作用を `lifespan` に集約し、テスト可能性を高める。

**完了条件（DoD）**
- `on_event` 依存を解消し、`lifespan` で同等挙動を担保すること。
- 起動/停止シーケンスの回帰テストが追加されること。
