# READMEレビュー（README.md / README_JP.md）

作成日: 2026-03-02
対象:
- `README.md`
- `README_JP.md`

## 総評
- 両READMEとも、プロジェクトの価値提案（ガバナンス重視）と主要サブシステム（Planner / Kernel / FUJI / MemoryOS など）の説明は明確です。
- セキュリティ記述は比較的充実しており、CORS・鍵管理・PIIへの注意喚起が含まれています。
- 一方で、英語版にMarkdown構文不整合と章構成の重複があり、可読性・保守性・運用時の誤解リスクがあります。

## 指摘事項（優先度順）

### 1. [High] README.md のコードフェンス終了記号が不整合
- 場所: `README.md` の「Pipeline mental model」セクション
- 事象: 開始が ```text なのに終了が ````（バッククォート4つ）になっている。
- 影響: Markdownレンダラによっては以降の表示崩れが発生し、読者が手順や注意事項を誤読する可能性があります。
- 推奨対応: 終了記号を ``` に統一する。

### 2. [Medium] README.md のセキュリティ章が二重化し、目次との対応が分散
- 場所: `README.md`
- 事象:
  - 途中に `## Operational Security Deep Dive`
  - 後半に `## Security Notes (Important)`
  が併存している。
- 影響: 参照導線が分断され、どちらを正とすべきか利用者が迷う。
- 推奨対応:
  - 章を統合し、目次リンク先を一本化する。
  - あるいは「概要（Quick）」と「詳細（Deep Dive）」の親子構造を明示する。

### 3. [Medium] README.md / README_JP.md 間で運用セキュリティ説明の粒度が微妙に異なる
- 事象: 両言語のセキュリティ表現は概ね一致するが、章構成と記述の配置が揃っておらず、将来的な同期漏れが起きやすい。
- 影響: 一方の言語のみ更新されることで、セキュリティ要件の伝達が言語依存になる。
- 推奨対応:
  - セキュリティ章を「共通テンプレート化」し、EN/JPで同一見出し構造を維持する。
  - 差分は翻訳表現のみに限定する。

## セキュリティ観点での警告（必読）
- **警告1**: READMEにある通り、`VERITAS_API_SECRET` へ短い値・既知値・プレースホルダを設定すると、HMAC保護が実質的に弱体化します。運用環境では32文字以上の高エントロピー値を必須化すべきです。
- **警告2**: `allow_credentials` を有効にした状態でCORSをワイルドカード運用すると、ブラウザ経由の不正アクセスリスクが高まります。許可オリジンは明示列挙に限定してください。
- **警告3**: TrustLog/Memoryへの保存データにPIIが混入する可能性があるため、保存前マスキング・アクセス制御・保持期間管理・保存時暗号化（要件次第）の組み合わせが必要です。

## 推奨アクション（短期）
1. `README.md` のコードフェンス不整合を修正。
2. 英語版セキュリティ章を統合し、目次と1対1で対応。
3. EN/JP READMEのセキュリティ章を同一アウトラインで再編成。
4. ドキュメントCI（markdown lint + link check）を追加し、構文崩れとリンク不整合を自動検出。

## 備考
- 本レビューはドキュメント評価のみであり、Planner / Kernel / FUJI / MemoryOS の実装責務には変更を加えていません。
