## CODEXで実施すべきベスト改善（結論）

**`Planner / Kernel / Fuji / MemoryOS` の責務境界をCIで自動強制し、境界単位の契約テストを標準化する。**

### 選定理由

- **再発防止**: 境界逸脱は機能追加時に静かに混入しやすく、人手レビューのみでは検知漏れが起きる。
- **安全性向上**: 境界が崩れるとFuji/監査があっても検証抜けが生じ、「安全側に倒れる」性質が弱まる。
- **開発速度向上**: PR段階で違反を即時検知できるため、後工程デバッグを削減できる。

### 実装方針（責務境界を越えない範囲）

1. `scripts/architecture/check_responsibility_boundaries.py` をCI必須ジョブ化。
2. 主要pipelineに契約テスト（正常系＋異常系）を追加。
3. 境界違反時にCIログへ修正ガイド（許可依存先・代替先）を出力。

### 受け入れ基準（CODEXレビュー観点）

- **境界順守**: `Planner / Kernel / Fuji / MemoryOS` を跨ぐ新規importが0件。
- **安全性**: 異常系テストFail時に処理が必ず安全側（deny/fallback/logging）へ遷移。
- **運用性**: 失敗ログが「境界違反/入力不正/権限不足」を機械分類可能。
- **生産性**: 境界違反修正に必要な情報がCIログ内で完結。

## セキュリティ警告（重要）

- **警告**: 境界検査を導入しても、秘密情報（APIキー、署名鍵、内部URL）の管理運用が弱い場合、攻撃面は残る。
- **警告**: LLM/Web入力経路では prompt/data injection が継続リスクであり、入力検証・サニタイズ・最小権限を同時に強化する必要がある。
- **警告**: 鍵ローテーションや公開設定監査が未徹底だと、署名基盤が存在しても信頼性は急速に低下する。
