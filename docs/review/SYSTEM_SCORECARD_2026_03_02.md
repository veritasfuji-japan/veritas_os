## 「CODEXで改善すべきベストな改善」

**結論: `Planner / Kernel / Fuji / MemoryOS` の責務境界をCIで自動強制しつつ、境界ごとの契約テストを同時に標準化すること。**

- 理由1（再発防止）: 境界逸脱は機能追加時に静かに混入しやすく、レビューだけでは検知漏れが発生する。
- 理由2（安全性）: Fujiや監査系があっても、境界越境で検証抜けが起きると「安全側に倒れる」挙動が壊れやすい。
- 理由3（開発速度）: 依存ルール違反をPR段階で即時に落とせれば、後工程のデバッグ時間を大幅に削減できる。

### 実装方針（責務境界を越えない範囲）

1. `scripts/architecture/check_responsibility_boundaries.py` をCI必須ジョブ化する。
2. 主要pipelineについて「正常系 + 異常系（timeout/invalid input/permission）」の契約テストを追加する。
3. 境界違反時はCIログに修正ガイド（許可依存先一覧）を出力し、修正時間を短縮する。

### セキュリティ警告

- **警告**: 境界検査を導入しても、秘密情報の取り扱い（APIキー、署名鍵、内部URL）に対する運用監査が弱いと、攻撃面は残る。
- **警告**: LLM入力経路では prompt/data injection が継続リスクであり、入力検証・サニタイズ・最小権限を同時に強化する必要がある。

## セキュリティ警告（重要）

- **警告**: ガバナンスや署名基盤があっても、実運用での鍵ローテーション・秘密情報管理・公開設定監査が崩れると、信頼性は急速に低下する。
- **警告**: LLM連携やWeb連携を含むため、入力汚染（prompt/data injection）に対する継続的防御（検証・サニタイズ・最小権限）が必須。
